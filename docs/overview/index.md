# 平台概览

本科生算力平台的核心使用方式是：在登录入口准备文件和环境，把计算任务提交给调度系统，由计算节点实际运行任务。理解这个模型后，很多看似零散的问题都会变清楚。

## 必须知道

- **登录节点**用于登录、整理文件、编辑脚本、安装轻量依赖和提交作业。不要在登录节点上训练模型、跑大规模仿真或长时间占满 CPU。
- **计算节点**用于真正运行计算任务。GPU 训练、大规模数值计算、长时间脚本都应该通过 Slurm 申请资源后在计算节点运行。
- **共享存储**让登录节点和计算节点能看到同一份用户文件。你在登录节点上传的代码和创建的环境，通常可以在计算任务中访问；具体路径和容量限制仍需以平台说明为准。
- **Slurm**负责排队和分配资源。你提交作业时，需要说明大致要多少 CPU、GPU、内存和运行时间。

## 推荐使用模型

最小闭环：

1. 在 GUI 或 Shell 中上传代码和数据。
2. 建立一个独立项目目录。
3. 写一个很小的测试脚本。
4. 用 `sbatch` 提交作业。
5. 用日志确认结果。

长期项目：

1. 每个项目单独建目录，固定代码、数据、日志、输出位置。
2. 每个项目使用独立环境，保留 `requirements.txt`、`environment.yml` 或安装记录。
3. 先跑 smoke test，再跑 benchmark，最后跑正式任务。
4. 所有正式任务都用脚本提交，避免只在交互式 Shell 里手动运行。

## 公平使用

平台是共享资源池。提交任务前请估算需求，先用小任务验证，再扩大资源。发现任务卡死、无限循环、日志没有进展或资源占用异常时，应及时取消任务。

不要把共享存储当个人网盘长期堆放无关文件。课程作业、实验结果和训练日志应定期整理，确认不再需要的大型中间文件可以删除或转存。

## 常见误解

- “能打开 Shell 就能直接跑训练”：不建议。训练应提交到计算节点。
- “申请越多资源越快”：不一定。CPU worker、内存和 GPU 数量都可能受程序瓶颈影响，需要测试。
- “浏览器关了任务就停”：批处理作业通常由调度系统运行，不依赖浏览器窗口；交互式任务和网页终端行为需要按平台实际说明确认。

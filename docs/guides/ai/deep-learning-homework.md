# 人工智能专业：提交深度学习作业训练任务

本文面向需要在本科生算力平台上完成深度学习课程作业训练任务的同学。目标不是覆盖平台全部能力，而是给出一条可以直接照着走的最小操作路径。

## 适用场景

适用于以下情况：

- 你已经拿到课程作业代码，需要在平台上跑训练
- 你需要申请 GPU 资源执行一次或多次训练任务
- 你希望训练过程在后台运行，并且可以查看日志

不在本文重点范围内的内容：

- 多机分布式训练
- 自定义镜像构建
- 大规模实验管理

## 建议的操作流程

对于大多数深度学习作业，建议按下面顺序操作：

1. 登录平台并进入自己的工作目录
2. 上传作业代码、数据集和依赖文件
3. 准备 Python 环境
4. 先在小规模数据或少量 epoch 上做一次快速验证
5. 编写并提交训练任务
6. 查看日志，确认训练正常开始
7. 训练结束后下载结果或整理实验记录

## 提交前需要准备什么

在提交训练任务前，建议先确认下面几项：

- 作业代码可以在本地正常运行
- 已经明确入口脚本，例如 `train.py`
- 已经明确依赖安装方式，例如 `requirements.txt` 或 `environment.yml`
- 已经明确训练输出目录，例如 `outputs/` 或 `checkpoints/`
- 已经明确这次训练需要的资源，例如 GPU 数量、显存、CPU、内存、预计时长

如果这些信息都不清楚，提交任务时最容易出现的问题是：

- 资源申请过大，长时间排队
- 环境缺依赖，任务启动后立即失败
- 输出路径混乱，训练完成后找不到结果

## 一个推荐的目录结构

建议把一次作业整理成类似结构：

```text
homework/
├─ code/
│  ├─ train.py
│  ├─ model.py
│  └─ requirements.txt
├─ data/
├─ outputs/
└─ logs/
```

这样做的好处是：

- `code/` 放代码和依赖说明
- `data/` 放训练数据
- `outputs/` 放模型和结果
- `logs/` 放任务日志

后续复现实验时更清晰。

## 推荐的训练任务提交流程

### 第一步：上传代码与数据

先将作业代码、配置文件、数据集上传到平台工作目录。

这一阶段建议重点确认：

- 路径是否正确
- 数据是否完整
- 训练入口文件是否存在

> 这里后续可以补充 GUI 上传方式和命令行上传方式。

### 第二步：准备运行环境

根据课程作业要求准备 Python 环境，并安装依赖。

常见做法包括：

- 使用平台预装环境
- 创建自己的虚拟环境
- 根据 `requirements.txt` 安装依赖

建议至少先手动执行一次下面这类检查：

```bash
python train.py --help
```

如果连帮助信息都无法正常输出，不要急着提交正式训练任务，应先把环境问题解决掉。

### 第三步：做一次快速冒烟测试

正式申请 GPU 长任务前，建议先进行一次小规模验证。例如：

- 只跑 1 个 epoch
- 只取少量样本
- 使用较小 batch size
- 只验证脚本能否启动并正常写日志

这个步骤能提前发现很多低级问题，例如：

- 数据路径写错
- 缺少依赖
- 输出目录没有权限
- 参数名写错

### 第四步：编写训练命令

将正式训练命令整理清楚，示意如下：

```bash
python train.py \
  --data-dir ../data \
  --output-dir ../outputs/exp1 \
  --epochs 50 \
  --batch-size 64 \
  --lr 1e-3
```

建议把训练输出单独写到一个实验目录中，避免多次运行互相覆盖。

### 第五步：提交训练任务

提交任务时，需要至少明确以下几类信息：

- 任务名称
- 运行命令
- 工作目录
- GPU / CPU / 内存需求
- 最长运行时间
- 日志输出位置

建议优先遵循“够用即可”的资源申请原则：

- 单卡能跑就不要申请多卡
- 短任务不要申请过长时间
- 小作业不要申请过多内存

这样更容易排队成功，也更符合课程作业场景。

> 这里后续可以分别补充：
>
> - 通过 GUI 提交训练任务
> - 通过命令行提交训练任务
> - 常见参数如何填写

### 第六步：查看日志与任务状态

任务提交后，需要重点看两件事：

1. 任务是否成功进入运行状态
2. 日志中是否已经出现正常训练输出

至少应检查：

- 是否成功加载数据集
- 是否成功识别 GPU
- loss 是否正常变化
- 是否有显存不足或依赖报错

如果任务刚启动就失败，优先检查：

- 工作目录是否正确
- 命令是否正确
- 环境是否激活
- 路径是否使用了平台上的真实路径

### 第七步：整理结果

训练完成后，建议至少保留以下内容：

- 训练日志
- 最终模型权重
- 关键指标
- 本次使用的命令或配置

如果课程作业需要写报告，这些内容后续都要用到。

## 面向课程作业的建议

### 资源申请建议

对于课程作业，通常优先考虑：

- 小规模、可复现
- 结果稳定
- 资源使用克制

不要默认把“更大的资源申请”当成“更好的实验方案”。对于作业任务，更重要的是：

- 能稳定跑通
- 能解释实验设计
- 能复现实验结果

### 实验管理建议

建议每次训练都记录清楚：

- 实验名称
- 模型结构
- 关键超参数
- 数据版本
- 最终结果

最简单的做法是为每次实验建立独立输出目录，并保留启动命令。

### 常见错误

课程作业中最常见的问题通常是：

- 直接提交正式长任务，没有先做冒烟测试
- 多次训练都写入同一个输出目录
- 忘记保存日志
- 申请资源明显超过实际需要
- 没有固定随机种子，结果难以复现
